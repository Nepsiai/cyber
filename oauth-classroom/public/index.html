Conceptos Clave de OAuth 2.0

Entiende los actores y elementos antes de ver el flujo completo

🧑

Resource Owner (Usuario)

Es el usuario dueno de los datos. Ejemplo: tu, cuando autorizas a una app a acceder a tu perfil de Google.

📱

Client (Aplicacion)

La aplicacion que quiere acceder a los datos del usuario. Ejemplo: Spotify pidiendo acceso a tu cuenta de Facebook.

🔐

Authorization Server

El servidor que autentica al usuario y emite tokens. Ejemplo: los servidores de Google cuando haces "Login con Google".

📦

Resource Server (API)

El servidor que tiene los datos protegidos. Acepta peticiones solo con un Access Token valido.

🎟️

Authorization Code

Codigo temporal y de un solo uso. Se intercambia por un Access Token. Viaja por el navegador (front channel).

🪙

Access Token

La "llave" que permite acceder a recursos protegidos. Tiene tiempo de expiracion y permisos (scopes) limitados.

🔄

Refresh Token

Token de larga duracion que permite obtener nuevos Access Tokens sin que el usuario tenga que volver a iniciar sesion.

🔑

Scopes (Permisos)

Definen que puede hacer la app. Ejemplos: read:profile, write:posts, read:email.

🔗

Redirect URI (Callback)

URL pre-registrada donde el Auth Server envia al usuario despues de autorizar. Previene ataques de redireccion.

🛡️

State Parameter

Valor aleatorio que previene ataques CSRF. El cliente lo envia y verifica que regrese identico en la respuesta.

📋

PKCE (Proof Key for Code Exchange)

Extension de seguridad que genera un code_verifier y code_challenge para prevenir intercepcion del authorization code.

🆔

OpenID Connect (OIDC)

Capa sobre OAuth 2.0 que agrega autenticacion. Devuelve un ID Token (JWT) con la identidad del usuario.
